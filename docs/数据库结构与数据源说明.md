## 三、核心数据库操作（[database.py](../database.py)）

### DatabaseManager 类的主要方法

| 方法 | 功能 |
|------|------|
| `get_session()` | 获取数据库会话（上下文管理器） |

| `get_paper_count(user_id, ...)` | 获取论文总数 |
| `get_statistics(user_id)` | 统计各主题数量、已审核/未审核数 |

| `update_paper_theme(...)` | 更新论文主题分类（人工审核） |
| `toggle_star(paper_id, user_id)` | 切换收藏状态 |
| `delete_all_papers(user_id)` | 删除用户所有论文 |
| `initialize_sample_data(user_id)` | 初始化示例数据 |
| `batch_reclassify_papers(user_id)` | 批量重新分类 |
| `export_papers(user_id, format)` | 导出 CSV/Excel |

### 核心查询逻辑

**主题过滤**：`theme_bucket` 字段精确匹配所选主题

**标签过滤**：
- **核心标签**：`core_matched_keywords` 字段使用 `LIKE` 模糊匹配
- **扩展标签**：`matched_keywords` 字段使用 `LIKE` 模糊匹配
- **联合过滤**：核心和扩展标签的筛选结果合并（OR 关系）

**关键词搜索**：`title` 字段使用 `LIKE` 模糊匹配

**数据隔离**：所有查询均需 `user_id` 参数，不同用户互不可见

| `Materials` | 材料与未来制造 | 14 | 13 | `#e67e22` |

| `Engineering` | 未来工程与社会系统 | 5 | 12 | `#3498db` |

**分类规则**：
1. 标题和摘要中匹配到某个主题的**核心关键词** → 直接归属该主题，置信度 0.9
2. 匹配到**扩展关键词**但未匹配核心关键词 → 归属该主题，置信度 0.7
3. 同时匹配多个主题 → 选择匹配数量最多的主题
4. 未匹配任何主题 → 归属 `Other`（其他）

**主题配置数据结构**：

### 接入方式说明

#### 1. API 接入（推荐，最稳定）
- **arXiv**：使用 `arxiv` Python 库，支持分类过滤和关键词搜索
- **OpenAlex**：调用 REST API `/works` 端点，支持标题/摘要搜索
- **PubMed**：调用 NCBI E-utilities API（`esearch` + `esummary`）

#### 2. RSS 订阅（稳定）
- **Science Daily**：`rss/top.xml` + 分类 RSS
- **MIT Technology Review**：`technologyreview.com/feed/`
- **TechCrunch**：`techcrunch.com/feed/`
- **通用 RSS**：通过 `feedparser` 解析任意 RSS/Atom feed

#### 3. 通用网页爬虫（可行，数据质量依赖目标网站）
- 使用 `BeautifulSoup` 解析 HTML
- 自动识别文章链接（`.html`、`/article/`、`/news/` 等模式）
- 支持关键词过滤
- 适用于：中关村论坛、中国科普博览、中国数字科技馆、Nature 首页等

### 关键词搜索机制

- **英文 API 数据源**（arXiv、OpenAlex、PubMed）自动将中文关键词翻译为英文
- **翻译映射表**位于 `crawler.py` 的 `KEYWORD_TRANSLATION`，覆盖 50+ 常见中英关键词
- **中文数据源**（科普网站、论坛等）直接使用原始中文关键词过滤
- **RSS 订阅源**在标题和摘要中匹配关键词

### 采集架构

```
CrawlerManager（并发调度）
```

## 八、采集逻辑详解

### 1. 核心采集流程

```
用户操作                   采集层                             数据层
   │                         │                                 │
   ├─ 选择数据源/输入关键词    │                                 │
   ├─ 点击"开始采集" ───────► CrawlerManager 初始化              │
   │                         ├─ _setup_crawlers() 配置采集器     │
   │                         ├─ crawl_all() 并发调度             │
   │                         │  ├─ ThreadPoolExecutor (4线程)    │
   │                         │  ├─ 每个 crawler.crawl() 独立执行   │
   │   ◄── 实时进度回调 ──────┤  ├─ progress_callback(进度, 消息) │
   │   ◄── 停止检测 ─────────┤  ├─ stop_check_callback()         │
   │                         │  └─ 异常隔离（单个失败不影响其他）  │
   │                         ├─ 数据格式化 format_paper_data()    │
   │                         └─ 批量入库 add_papers() ──────────► papers 表
   │                                                                 │
   └─ 实时查看结果 ◄──────────── 刷新仪表盘/数据浏览 ◄──────────────┘
```

### 2. 实时进度反馈机制

这是采集系统的核心特性，用户发起采集后能实时看到每个数据源的进度。

#### 2.1 回调函数接口

```python
# CrawlerManager.crawl_all() 的两个回调参数：
def crawl_all(
    self,
    progress_callback: Callable[[float, str], None] = None,    # 进度回调
    stop_check_callback: Callable[[], bool] = None              # 停止检测回调
) -> Dict[str, int]:
```

| 参数 | 类型 | 说明 |
|------|------|------|
| `progress_callback(progress, message)` | `(float, str) -> None` | 每完成一个数据源调用一次，`progress` 范围 0~1，`message` 为当前完成的数据源名 |
| `stop_check_callback()` | `() -> bool` | 每次回调前调用，返回 `True` 则停止后续采集 |

#### 2.2 进度通知流程

```
ThreadPoolExecutor 并发执行
         │
         ├─ crawler_1 完成 ──► progress_callback(1/总采集数, "已完成: arXiv")
         ├─ crawler_2 完成 ──► progress_callback(2/总采集数, "已完成: PubMed")
         ├─ crawler_3 完成 ──► progress_callback(3/总采集数, "已完成: 中关村论坛")
         └─ ...每完成一个通知一次
```

- `progress = completed / total_crawlers`（已完成数 / 总采集器数）
- 每个采集器执行前后会检查 `stop_check_callback()`，用户可随时停止
- 停止时调用 `executor.shutdown(wait=False)`，不等待正在执行的任务

#### 2.3 Streamlit 进度展示方式

```python
# 在 Streamlit 前端中的典型用法（tasks.py 或 app.py 中）：
progress_bar = st.progress(0, text="准备开始采集...")
status_text = st.empty()

def on_progress(progress: float, message: str):
    """实时更新进度条和状态文本"""
    progress_bar.progress(progress, text=message)
    status_text.text(message)

# 传入回调开始采集
results = manager.crawl_all(
    progress_callback=on_progress,
    stop_check_callback=lambda: st.session_state.get('stop_crawl', False)
)
```

#### 2.4 最终结果反馈

采集完成后返回每个数据源的采集结果：

```python
# crawl_all() 返回值结构
{
    "arXiv": {"count": 15, "papers": [...]},        # 获取了 15 篇
    "PubMed": {"count": 0, "papers": []},            # 采集失败或无结果
    "中关村论坛": {"count": 10, "papers": [...]},
    # ...每个数据源一条记录
}
```

### 3. Crawler 类体系

```
CrawlerBase (基类)
├── 属性: source_name, max_results, search_query, session
├── _create_session() → 带重试机制的 requests.Session
│   ├── Retry: total=3, backoff=1, status_forcelist=[429, 500, 502, 503, 504]
│   └── HTTPAdapter 挂载 http:// 和 https://
├── crawl() → [Dict]  (抽象方法，子类必须实现)
└── close() → 关闭 session

子类              | 数据源       | 接入方式   | 主要依赖
──────────────────|─────────────|-----------|-------------------
```

### 4. 统一数据格式

每个 Crawler 的 `crawl()` 方法返回原始 dict 列表，最终由 `format_paper_data()` 统一格式化：

**原始 dict 字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `source_type` | str | 数据源标识，如 `arXiv`, `PubMed`, `Nature` |
| `title` | str | 文章标题（最多 500 字符截断） |
| `abstract` | str | 摘要或正文简介 |
| `url` | str | 原文链接 |
| `published_date` | date/str | 发布日期 |
| `signal_type` | str | `paper`（学术论文）或 `news`（新闻/科普） |
| `field` | str | 学科领域/分类 |

**`format_paper_data()` 处理流程：**

1. **日期标准化**：`parse_date()` 尝试 10+ 种日期格式解析
2. **文本清洗**：`clean_text()` 去除 HTML 标签、多余空白，超长文本截断
3. **URL 补齐**：相对路径转为绝对 URL
4. **内容哈希**：基于 `(标题 + 来源)` 生成 SHA256 的 `content_hash`，用于去重
5. **默认值填充**：缺失字段补充默认值

### 6. 内容去重机制

#### 6.1 哈希生成

```python
# 基于 标题 + 来源URL 生成 SHA256 哈希
paper['content_hash'] = hashlib.sha256(
    f"{title}|{url}".encode()
).hexdigest()
```

#### 6.2 去重流程

```python
# DatabaseManager.add_papers() 中的去重逻辑：
def add_papers(self, papers: List[Dict], user_id: int) -> Tuple[int, int]:
    new_count = 0
    duplicate_count = 0
    for paper in papers:
        # 检查 content_hash 是否已存在
        existing = session.query(Paper).filter(
            Paper.content_hash == paper['content_hash'],
            Paper.user_id == user_id
        ).first()
        if existing:
            duplicate_count += 1
            continue
        # 插入新记录
        session.add(Paper(**paper_data, user_id=user_id))
        new_count += 1
    session.commit()
    return new_count, duplicate_count  # (新增数, 重复数)
```

### 7. 并发与异常处理

#### 7.1 ThreadPoolExecutor 并发

```python
with ThreadPoolExecutor(max_workers=4) as executor:
    futures = {executor.submit(run_crawler, crawler): crawler for crawler in self.crawlers}
    
    for future in as_completed(futures):
        # 每个采集器完成时处理结果
        source_name, count, papers = future.result()
        # 更新进度...
```

- **max_workers**: 4（可配置）
- **调度策略**: 所有采集器同时提交，谁先完成谁先通知
- **异常隔离**: 单个采集器异常不影响其他采集器

#### 7.2 超时与重试

| 机制 | 配置 |
|------|------|
| Session 重试 | 3 次，退避因子 1s |
| HTTP 超时 | 30s（通用），15s（RSS） |
| API 特殊处理 | arXiv 节流等待 3s，OpenAlex 429 自动退避 |

#### 7.3 停止采集机制

```python
def stop(self):
    """停止采集（线程安全）"""
    self._stopped = True

# 在 crawl_all 中的两处检查点：
# 1. 提交任务前检查
if self._stopped or (stop_check_callback and stop_check_callback()):
    return  # 跳过该采集器
# 2. 每完成一个采集器后检查
if progress_callback:
    if stop_check_callback and stop_check_callback():
        executor.shutdown(wait=False)  # 强制停止剩余任务
        break
```
